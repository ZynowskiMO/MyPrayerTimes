# CLAUDE.md — working agreement for PrayerTimes

## Who you're working with

The repo owner is the **Product Owner / IT Manager, not a programmer**.
Do not ask for or expect line-by-line code review. Explain in outcomes and
trade-offs, not in diffs.

## Source of truth

- `docs/SPEC.md` (v2.0) defines what to build and the phased delivery.
- `docs/ADR-0002-local-calculation.md` defines the current architecture and why.
- `docs/ADR-0001-time-source.md` is **Superseded** — kept for history only.
  Do not follow it; ADR-0002 governs.
- If something is ambiguous or contradicts these, STOP and ask before coding.
- Do not silently expand scope. Out-of-scope items in SPEC stay out.

## Phasing

- Build strictly in phase order (SPEC §3). Do not start a phase until the
  previous phase's exit criterion is met and confirmed by the owner.
- Phase 1 is the calculation engine. Nothing visual ships until its output
  matches adhan-js reference values within tolerance.

## Build discipline

- **Checkpoint every ~50 generated lines.** Stop, summarize, wait.
- **Write an ADR before any architectural decision** not already covered by
  ADR-0002 (new dependency, changed engine behavior, new persistence model).
- Confirm before destructive actions (deleting files, force-push,
  overwriting fixtures). State the risk and wait for explicit "yes".

## Change reporting format

```
PROMJENA:      <what changed, technical>
KORISNIK VIDI: <what the user observes in-game / when running tools>
RIZIK:         <what could go wrong; "none" only if truly none>
```

## Correctness (non-negotiable for a public prayer-time addon)

- Port the math from adhan-js (batoulapps/adhan-js). Do NOT write prayer-time
  math from memory.
- Commit verification fixtures: date → expected times generated by adhan-js.
- A change to the engine is not "done" until fixtures still pass within
  ±1 minute.

## Licence

- adhan-js is MIT. Keep its licence text in `THIRD_PARTY_LICENSES.md` and
  credit batoulapps in README and the CurseForge listing.

## Language

- All code, comments, commit messages, and docs in **English**.
- Conversational replies to the owner may be in Bosnian/Croatian/Serbian.

## Code expectations

- Handle missing-data and bad-input cases per function.
- Addon: use only API stable on BOTH Retail and Classic (SPEC §6).
- Verify current `## Interface:` numbers at build time; don't trust memory.

## Git

- Small, focused commits per checkpoint.
- Commit subject: short imperative; body carries the PROMJENA/RIZIK note.
